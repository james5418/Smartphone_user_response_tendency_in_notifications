Smartphone user’s response tendency in notifications

Introduction

Objectives

Smartphone notifications are indispensable to users, but they often demand users’ attention at inappropriate moments. As a result, whether the user will actually attend to a notification soon is very important. In order to solve this problem, I intend to attain the goals below.

  • Predict whether a user is likely to attend to a smartphone notification within a certain time period
  • Knowing what kind of notification modalities will attract users
  • Construct a non-personalized model for each smartphone user

About the project

  • Modeling and predicting user responsiveness to smartphone notification using machine learning classifiers
  • Focus on the response time from a notification
  • Refrain from using privacy-sensitive features due to privacy concerns
  • Prediction is based on physical features (sound, LED, vibration) instead of content (title, message)
  • Smartphone users may have different preferences for notification display, and a non-personalized model can be applied to each user

Data Collection

Dataset description

  • This dataset is from GitHub
  • Contain details of 176,195 logged notifications received on the Android devices of 14 users, in the period between 2018-05-07 and 2018-06-05
  • Include both notification features and device features
    • notification features: time posted, time removed, sound, LED, vibration, etc.
    • device features: ringer mode, interactive, screen state, etc.
  • Data comes from user’s behavior with notifications from any app and the Android OS itself

Data collection method

  • Android OS allows application programmers to design specific notification modalities for the apps, so we can get this information from Android API
  • Get the log data of notifications from Android API calls NotificationListenerService
  • Use this API to build a simple logging application that can trace all notification activities in the background

Preprocessing

Data clean-ups

  1. Since the dataset is a sql file, I manually remove SQL syntax in it and then convert it to txt file
  2. Drop some columns that are useless for the training model, such as id, notification_id, flag
  3. Filter out any notification that its timeremoved equals to 0
  4. Derive a feature response_time as the label by subtracting the dismissal time(timeremoved) of the notification from the issuing time(timeposted) of the notification
  5. On some devices the OS would generate many notifications which it issued and dismissed at the same time, so any notifications that had a response_time equal to 0 were also filtered out
  6. Drop the features timeposted and timeremoved

Feature transform

Turn the problem of modeling user attentiveness into a class-prediction problem.

  1. Compute the mean of response_time as a threshold (\(575s\)) for classifying the user as having “high” or “low” attentiveness to the notification they receive
  2. Label each notification by the threshold
    • if response_time < threshold, then y=1 for high attentiveness
    • if response_time >= threshold, then y=0 for low attentiveness
  3. Drop the feature response_time

If the model predicts that a notification is high attentiveness(y=1), then we consider the user will tap into this notification, and vice versa.

Transform data format

  1. Divide the dataset into y(attentiveness) and X
  2. Encode X by OrdinalEncoder() because the feature application_name is string type
  3. Suuffle the data

Models

Using SciKit-Learn package

  • Naïve Bayes (with Laplace smoothing)
  • Random Forest (10 trees)
  • SVM (RBF kernel with gamma=0.7)
  • MLPClassifier (hidden_layer_sizes=(200,))

Results

  • For each model, a 10-fold cross-validation was performed
  • Train the model with and without user_id

With user_id

Without user_id

  • All of the models have acceptable performance, and the SVM model makes the most precise prediction, it achieves approximately \(90\%\) accuracy.
  • There is no significant difference between using user_id feature and not using user_id feature, thus we can speculate that this feature is not a key point to affect the result. However, there are more factors that need to be considered.

Conclusion

  • This model shows the feasibility of using notification logs to predict whether a user will engage in the notification within a certain time period with good accuracy
  • user_id doesn’t seem to improve performance significantly, hinting that personalization of predictive models might not be necessary
  • Knowing how to adjust notification settings to attract users to tap on notifications and increase users’ willingness to enter the app

Application

Using this model, we can implement a service that automatically predicts and updates users’ availability to attend to a notification by viewing their notification settings. And then send the notifications at a time when they are likely to see the notifications. In this way, notifications might not cause potential disruption to their tasks.

Also, this result can be applied in many applications.

  • Help developers determine the most suitable notification form according to different purposes
  • Help applications to catch user’s attention by push notifications
  • Improve the experience of smartphone users on notifications
